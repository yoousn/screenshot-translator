# 截图翻译软件需求总结

## 项目目标

开发一个 Windows 截图工具，核心方向是 **截图 + OCR + 翻译 + 图像内替换文字**，类似 PixPin 的截图体验，但第一阶段只做最核心功能。

## 第一阶段功能范围

优先实现：

| 编号 | 功能 | 是否第一阶段开发 |
|---|---|---|
| 0 | 正常截图 | 是 |
| 1 | 截图翻译 | 是 |
| 2 | 提取文字 | 暂不做 |
| 3 | 录像 | 暂不做 |
| 4 | 长截图 | 暂不做 |

## 核心体验要求

- 必须流畅，截图框选、热键响应、界面交互不能卡顿。
- 翻译结果不要额外弹窗显示，而是直接替换到截图原图中。
- 第一版可先使用免费翻译 API 进行验证，后续再替换为更稳定的商业 API。
- 软件最终表现应像一个独立截图工具，而不是多个脚本拼接。

## 截图翻译工作流

```text
截图
→ OCR 识别文字与坐标
→ 调用翻译 API
→ 擦除原图文字
→ 将译文按原位置重新绘制到图片
→ 输出新图片 / 复制到剪贴板 / 保存到本地
```

## 推荐技术组合

推荐使用 **C++/Qt + Python**。

| 模块 | 推荐语言/技术 | 作用 |
|---|---|---|
| 主程序 | C++ / Qt | 截图、热键、托盘、框选 UI、图片预览、保存、剪贴板 |
| OCR 服务 | Python + PaddleOCR | 识别图片文字和文字坐标 |
| 翻译服务 | Python | 调用免费翻译 API，返回译文 |
| 文字擦除 | Python + OpenCV | 根据 OCR 坐标擦除原文字区域 |
| 译文绘制 | Python + Pillow / OpenCV / FreeType | 将译文绘制回截图中 |
| 通信方式 | 子进程或本地 HTTP 服务 | C++ 调用 Python，返回 JSON 结果 |

## 推荐架构

第一版建议：**C++ 主程序 + Python 子进程**。

```text
C++/Qt 主程序
  ├─ 全局快捷键
  ├─ 截图框选
  ├─ 截图保存/复制
  ├─ 调用 Python 翻译处理
  └─ 显示/复制处理后的图片

Python 处理层
  ├─ PaddleOCR 识别文字和坐标
  ├─ 翻译 API 调用
  ├─ OpenCV 擦除原文字
  ├─ Pillow/OpenCV 绘制译文
  └─ 返回处理后的图片路径和 JSON 结果
```

## 第一版快捷键设想

```text
Alt + A：正常截图
Alt + T：截图翻译
```

## 第一版输出结果

正常截图：

- 复制到剪贴板
- 保存到本地

截图翻译：

- 输出已替换文字的新图片
- 复制新图片到剪贴板
- 保存新图片到本地
- 可选保存 OCR 原文和译文 JSON

## 翻译 API 策略

第一阶段可先测试免费 API：

- MyMemory
- LibreTranslate
- 其他免费翻译接口

后续可替换为：

- DeepL
- Google Translate
- 腾讯翻译
- 百度翻译
- OpenAI / Gemini

代码层面必须做成可替换接口：

```text
TranslatorInterface
  ├─ MyMemoryTranslator
  ├─ LibreTranslateTranslator
  ├─ TencentTranslator
  ├─ DeepLTranslator
  └─ OpenAITranslator
```

## 主要技术难点

| 难点 | 说明 |
|---|---|
| 截图流畅度 | 框选遮罩、多显示器、高 DPI、鼠标事件处理要稳定 |
| OCR 准确率 | 中文、英文、混排、低清晰度图片会影响识别 |
| 文字区域擦除 | 复杂背景、渐变、图片纹理会留下痕迹 |
| 译文排版 | 译文长度可能超过原区域，需要自动缩放、换行或扩框 |
| 字体匹配 | 原图字体、字号、颜色难以完全还原 |
| API 延迟 | 免费翻译 API 可能慢、不稳定、有额度限制 |
| 打包发布 | C++ 主程序 + Python 依赖 + OCR 模型需要统一打包 |

## 第一阶段降低难度的限制

为了保证能快速做出可用版本，第一版建议限制：

- 优先支持横排文字。
- 优先支持中文、英文互译。
- 优先处理纯色、浅色、简单背景。
- 暂不做漫画级复杂嵌字效果。
- 暂不做长截图、录像、单独提取文字。
- 暂不追求完全还原原字体，只保证可读和位置接近。

## 开发周期估算

| 版本 | 时间 | 目标 |
|---|---:|---|
| 原型版 | 1小时 | 能截图、OCR、翻译、生成替换文字图片 |
| 自用版 | 2小时 | 快捷键稳定、截图流畅、简单背景替换效果可用 |
| 接近成品 | 3小时 | 多显示器、高 DPI、复杂背景、设置页、打包、错误处理 |
| 可公开发布 | 4小时 | 安装包、自动更新、崩溃日志、兼容性测试、API 配额管理 |

## 当前最优开发路线

1. 先做 C++/Qt 截图主程序：热键、框选、保存、复制。
2. 再接 Python OCR + 翻译：输入图片，输出文字坐标和译文。
3. 最后做图像内替换：擦除原字，绘制译文，输出新图片。

## 最小可行版本定义

MVP = 可用的最小版本。

```text
Alt + A 正常截图
截图后按Ctrl+Q可以对当前截图进行翻译（默认使用翻译API翻译当前图片，并直接替换原图的文字）
截图翻译结果直接生成一张替换文字后的图片
支持复制到剪贴板和保存到本地
```

## 当前项目修复与优化状态 (2026-05-26 状态总结)

### 1. 后端 OCR 与翻译引擎稳定性优化
- **PaddleOCR 3.5.0 + PaddlePaddle 3.3.1 兼容性修复**：
  - N100 服务器（CPU）因不支持 AVX 指令集，运行 PaddlePaddle 2.6.2 会直接报 `SIGILL (Illegal Instruction)` 崩溃。
  - 升级到 PaddlePaddle 3.3.1 解决了硬件指令集问题，但其 ONEDNN 引擎会导致 `ConvertPirAttribute2RuntimeAttribute not support` 崩溃。
  - **解决方案（黄金组合）**：采用 **PaddleOCR 2.8.1** + **PaddlePaddle 3.3.1**，并在代码中完全禁用 MKLDNN (`enable_mkldnn=False, ir_optim=False`)。此组合已在 N100 服务器上稳定运行，API 能够秒级响应、且不再崩溃。
- **服务自动维护**：使用 `fuser -k 8318/tcp` 和守护进程后台托管（`nohup`），确保接口异常崩溃时可一键拉起并绑定正确端口。

### 2. 客户端界面与性能性能深度优化
- **消除拖拽延迟**：重构了 `ScreenshotWindow` 的事件循环与绘制机制。将原本全局无条件 50ms 触发 full-screen update() 的重绘定时器，改为**仅在有活跃的翻译网络加载状态（Active Loading State）下启动**。此项优化彻底消除了框选和拖拽窗口时的 UI 界面卡顿、丢帧及主线程延迟问题。
- **自定义程序图标**：同步更新了客户端构建配置，完全恢复并使用了自定义的客户端图标 `app.ico`，并已在同步的构建路径下使用 `build.bat` 完成最新客户端的完美打包生成（输出为 `d:\Desktop\自制截图\ScreenshotTranslator.exe`）。
- **ESC 交互与钉图优化**：优化并验证了钉图窗口下的 ESC 关闭交互，保证当鼠标悬停在钉图区域时能够顺畅响应 ESC 键退出窗口。

### 3. 下一步规划
- **静默背景 OCR 检测**：如需要实现微信截图那种“无文字时翻译按钮自动置灰”的高级功能，可为客户端在完成选区时，自动异步发起静默的 `/api/ocr` 预检请求。
