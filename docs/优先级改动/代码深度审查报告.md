# YsnTrans 截图翻译器 · 代码深度审查报告（逐行通读 · 极致彻底版）

> 仓库：github.com/yoousn/screenshot-translator（分支 main，公开）
> 审查方式：通过 raw / api 接口逐文件读取源码，覆盖服务端 Python、Tauri/Rust 后端核心、前端 React/TS 核心逻辑。
> 本版在前几版基础上，补充了 OCR 子系统、录制子系统、原生捕获核心、前端全部核心 hooks 与 utils 的逐行通读结果。

---

## 一、项目概述

这是一个 Windows 桌面截图翻译 + 录屏工具，由三部分组成：

- **server/**：Python（FastAPI + uvicorn）翻译中转服务，聚合 Google / 百度 / DeepL / 大模型（new-api）等翻译通道。
- **src-tauri/**：Rust + Tauri 桌面后端，负责全局快捷键、截图捕获（GDI / DXGI / WGC）、OCR（RapidOCR ONNX）、录屏（ffmpeg）、窗口管理、配置存储、历史记录。
- **tauri-client/**：React + TypeScript + Antd 前端，负责截图叠加层交互、标注、滚动截图拼接、录制控制、设置面板、OCR/翻译结果渲染。

整体架构清晰、工程化程度高，但存在若干**真实功能性 Bug**、**安全隐患**、**性能瓶颈**，以及**大量未启用的诊断/脚手架代码**进入了发行构建。

## 二、严重 Bug（按严重程度排序）

| 编号 | 位置 | 问题 | 影响 | 建议 |
|---|---|---|---|---|
| **B1** | `rapid_ocr/worker.rs` + `mod.rs` | 常驻 OCR worker 的 `read_line` 读取**没有超时**，且 `with_rapid_ocr_worker` 在整个 OCR 请求期间持有全局 `RAPID_OCR_WORKER` 互斥锁。上层 `run_local_ocr` 用 `tokio::time::timeout` 包裹 `spawn_blocking`，一旦超时，阻塞线程被丢弃但仍在后台持锁运行 → **此后所有 OCR 永久卡死**。 | 致命：单次 OCR 卡住会导致功能整体不可用，且无法自愈 | 给子进程读写加超时；超时后**杀掉并重建** worker 进程，而不是丢弃线程；锁粒度细化 |
| **B2** | `hotkeys.rs` | 截图与翻译两个全局快捷键共用一个 450ms 防抖窗口（共享时间戳）。 | 快速连按或两个快捷键交替时会被误吞 | 每个快捷键独立防抖时间戳 |
| **B3** | `useScreenshotRecording.ts` `startRecording` | 录制选项先由 UI 读取 `fps`/`resolution`，随即被 `const normalizedOptions = { ...options, fps: 30, resolution: "1080p" }` **硬编码覆盖**。 | 用户在界面设置的帧率/分辨率被静默忽略，永远只录 30fps/1080p | 移除硬编码覆盖，使用用户实际选项 |
| **B4** | `hotkeys.rs` `parse_hotkey` | 解析快捷键时按 `+` 和 `-` 同时切分，导致包含 `-`（减号/Minus 键）的组合键被错误拆分。 | 含减号的自定义快捷键无法正确注册 | 仅按 `+` 分隔，单独处理 `-` 作为键名 |
| **B5** | `process_manager.rs` / `recording_commands.rs` | 录制热键 `RECORDING_HOTKEY` 在后端被硬编码，未跟随用户配置。 | 用户无法自定义录制快捷键 | 纳入配置体系 |
| **B6** | `app.py` | uvicorn 以 `reload=True` 启动（开发模式）。 | 生产环境下文件监听 + 双进程，资源浪费、行为不稳定 | 生产用 `reload=False` |
| **B7** | OCR 批处理（`localOcrTranslate.ts` / `translator.py`） | 多块文本按换行拼接后整体翻译，再按行切回；当翻译结果换行数与输入不一致时，**行错位**。 | 多段文本翻译串行/错位 | 用稳定分隔标记或逐块翻译并核对计数 |
| **B8** | `mergeRetryTranslations`（`ocrTranslationRequest.ts`） | 重试合并用 O(n²) `findIndex` 匹配。 | 大量文本块时性能退化（功能正确） | 用 Map 建索引 |

## 三、安全问题

| 编号 | 位置 | 问题 | 风险 | 建议 |
|---|---|---|---|---|
| **S1** | `security.py` / `app.py` | SSRF 防护对目标 URL 先解析 DNS 校验再请求，存在 **DNS-rebinding TOCTOU**（校验与实际连接之间 DNS 可变）；且大模型中转地址 `base_url` 由客户端任意指定。 | 服务端被诱导访问内网/元数据地址 | 校验后锁定已解析 IP 再连接；出网走固定解析；维护内网网段黑名单 |
| **S2** | 多处 `println!` / 日志 | `clientToken`、`x-api-key`、API Key 等敏感值可能进入日志/错误信息。 | 凭证泄漏 | 统一脱敏；`useSettingsController.ts` 的 `PRIVATE_TRANSLATION_ERROR_PATTERN` 脱敏思路应推广到后端 |
| **S3** | `app.py` | 翻译接口无速率限制、无请求体大小上限。 | 可被刷量 / 大包打满内存 | 加限流与 body 上限 |
| **S4** | `recording_process/ffmpeg_installer.rs` | `download_ffmpeg_release` 有 URL 白名单（BtbN/FFmpeg-Builds + objects.githubusercontent）与 `.zip` 校验，但**下载的二进制无校验和/签名验证**；且整包 zip 先全量读入内存（`Vec<u8>`）再落盘。 | 上游/镜像被污染时会执行未校验的 ffmpeg.exe；大包占内存 | 加 SHA256 校验；流式落盘 |
| **S5** | `process_manager.rs` | 剪贴板复制走 `powershell Set-Clipboard -LiteralPath`，音频设备名作为独立 ffmpeg 参数传递。 | 已确认**无 shell 注入**（参数化 + `''` 转义），此项为安全确认项 | 维持现状 |

> zip 解压已做 zip-slip 防护（固定目标路径），无路径穿越风险。

## 四、性能问题

| 编号 | 位置 | 问题 | 建议 |
|---|---|---|---|
| **P1** | `translationMemory.ts` `lookupLocalTranslation` | 每命中一个文本块就 `readStore()`（整盘 JSON.parse + TTL 过滤）**且** `writeStore()`（整盘排序 + slice + stringify），上限 1000 条。一次截图多块 → O(N×存储量)。 | 内存缓存 + 批量/节流落盘 |
| **P2** | `screenshot_commands.rs` | 全屏 RGBA 帧每次截图 `clone()` 2~3 次（rgba 守卫 + `encode_and_store_fullscreen_png` + PostSharedBuffer），4K≈33MB/份。 | 复用同一缓冲，减少克隆 |
| **P3** | `screenshot_commands.rs` `remember_image_save_directory` | 每次保存都**读取 + 解析 + 美化重写整个 config.json**。 | 仅在变化时写；后台异步写 |
| **P4** | `config_store.rs` | 按 key 逐次读盘（每个配置项一次磁盘读取）。 | 一次性加载并缓存 |
| **P5** | 截图热路径 | 大量 `println!("[screenshot-perf]…")` / `[baseline]` / `[window-trace]` / `[screenshot-trace]` 调试日志，前端还通过 `invoke("log_screenshot_perf")` 在交互热路径打点。 | 发行版降级或编译期关闭 |
| **P6** | RapidOCR | 冷启动加载 ONNX 模型约 340MB 内存、首次较慢。 | 懒加载 + 预热（已有预热）；考虑量化模型 |
| **P7** | 翻译候选排序 | LAN 优先（~465ms）对比公网（~1s），对冲请求错峰 700ms；逻辑合理但每块都重排候选。 | 缓存候选顺序 |
| **P8** | `detectionCandidates.ts` | 视觉边界检测对四个方向、四档 span 做逐像素边缘采样，鼠标交互时触发。 | 仅在无原生候选时启用（已做），可加节流 |

## 五、代码质量 / 可维护性

- **Q1（重大架构问题）：DXGI/WGC GPU 捕获是“已交付但不可达”的脚手架。** `capture_router.rs` 的 `resolve_capture_route` **恒定返回 `Fallback → ExistingCpu`**：任何非 CPU 偏好都被解析为 `GpuBackendPlaceholder`/`GpuBackendUnavailable`。但 `dxgi_capture.rs` 里有**真实可用的实现**（`DxgiDuplicationSession::open`、`acquire_next_frame`、500ms 首帧重试预算/50ms 单次、回读），`dxgi_readback.rs` 也很扎实（`checked_mul` 处理 pitch/溢出、BGRA→RGBA 交换）——**但路由永远不会选中它**。常量 `DXGI_PLACEHOLDER_REASON` = “native API wiring is intentionally pending”。与此同时，数十个 `*_probe.rs` / `*_smoke.rs` / `*_contract*` / `*_acceptance.rs` / `*_diagnostics_json.rs` 仍被编译进发行版。
- **Q2**：存在大量诊断/探针命令注册进 release（`screenshot_wgc_diagnostic_commands.rs`、`screenshot_wgc_selected_output_diagnostic_commands.rs` 41KB、`screenshot_dxgi_diagnostics_json.rs` 等），增加体积与攻击面。
- **Q3**：超大单文件 —— `screenshot_commands.rs` 约 **229KB**、`window_lifecycle.rs` 约 54KB。难以维护，建议拆分。
- **Q4**：1 字节占位 stub 文件（`ffmpeg_dependency.rs`、`file_commands.rs`）残留。
- **Q5**：重复的 `#[cfg(target_os = "windows")]` 包裹、手写 Win32 FFI（`text_source.rs` UIA 全树 `FindAll(TreeScope_Subtree)`、`recording_overlay.rs` 每次显示 `RegisterClassW` 却不注销）。
- **Q6**：约 17 处 Rust 未使用导入告警；前端多处 `config: any`、宽松 `any`。
- **Q7**：**UI 文案中英混杂**。`useScreenshotRecording.ts` 同时存在中文（“请选择要录制的窗口…”）和英文（“Please select a recording area first”、“No recordable window detected”、“Failed to enter recording mode”）；`useScreenshotLoader.ts` 的失败提示是英文。建议全部走 i18n。
- **Q8**：前端通过 localStorage + 事件传递录制会话 payload，并有 600ms 兜底 emit（`recordingWindows.ts`）——可用但属竞态规避式写法，建议改为创建即带参或可靠握手。
- **Q9**：默认大模型名 `"gemini-3.5-flash"`（`useSettingsController.ts`）为不存在的型号名，易误导。

## 六、逐文件审查结论（本轮新增 / 复核）

### 服务端（Python）
- `app.py`：FastAPI 路由 + uvicorn 入口；`reload=True`（B6）；无限流/无 body 上限（S3）。
- `translator.py`：多通道聚合；批量按行拼接翻译（B7 风险）。
- `security.py`：SSRF 校验（S1 DNS-rebind TOCTOU）。
- `config.py` / `translation_prompt.py`：配置与提示词常量，正常。

### Tauri/Rust 后端
- `lib.rs`（25KB）：命令注册中枢，注册了大量诊断命令（Q2）。
- `hotkeys.rs`（10.6KB）：B2 共享防抖、B4 `-` 拆分、B5 录制热键硬编码。
- `config_store.rs`：P4 按 key 读盘。
- `app_paths.rs`：`app_data_dir()` → `%LOCALAPPDATA%/ScreenshotTranslator`，正常。
- `history_commands.rs`：历史默认上限 100 条 / 2MB（可配 10–5000 / 64KB–100MB），正常。
- `text_source.rs`：UIA `collect_text_elements(hwnd, 240, 90)` 全子树遍历，开销较大（Q5）。
- `window_lifecycle.rs`（54KB）/`window_targets.rs`：窗口生命周期与目标枚举，体量过大（Q3）。
- `screenshot_commands.rs`（229KB）：截图主编排；P2 多次克隆、P3 保存重写 config、P5 热路径日志。
- `screenshot_shared_buffer.rs` / `recording_overlay.rs` / `diagnostics.rs`：共享缓冲 / 录制边框 / 诊断；`recording_overlay.rs` 重复 `RegisterClassW`（Q5）。
- **rapid_ocr/**：`mod.rs`（OCR 编排，B1 超时丢线程）、`runner.rs`、`worker.rs`（B1 阻塞读无超时 + 持全局锁）。
- **recording_process/**：`process_manager.rs`（录制进程管理，TOCTOU 已用二次加锁+kill 处理，剪贴板/参数安全 S5）、`ffmpeg_installer.rs`（S4 无校验和、全量入内存）、`device_detector.rs`（音频/设备探测，正常）。
- **screenshot_native/**：`capture.rs`、`capture_router.rs`（Q1 恒回退 CPU）、`dxgi_capture.rs`（真实实现但不可达）、`dxgi_readback.rs`（扎实，溢出防护到位）；其余 `*_probe/_smoke/_contract/_acceptance/_session/_output/_frame` 多为脚手架/测试（Q2）。

### 前端（React/TS）
- `App.tsx`：纯路由 + 布局壳；默认快捷键提示硬编码 `"Alt+A"`；正常。
- `useScreenshotLoader.ts`（928 行）：截图图像加载（base64/shared buffer 双路）、1500ms 超时、`recoverPreShowDrag` 轮询 48×16ms；失败提示为英文（Q7）；热路径 baseline 打点较多（P5）。
- `useScreenshotInteraction.ts`（923 行）：指针交互、选区/标注绘制；使用 `requestAnimationFrame` 调度（良好）；交互热路径有 `invoke("log_screenshot_perf")` 打点（P5）。
- `useScreenshotRecording.ts`：**B3 帧率/分辨率被硬编码覆盖**；中英混杂（Q7）；`[screenshot-trace]` 日志残留。
- `useScrollCapture.ts`：滚动截图，`setInterval` 760ms 自动滚动 + 抽样像素差拼接（启发式，可能错位）；上限 30 帧；逻辑合理。
- `useSettingsController.ts`：设置保存/通道测试；LAN→公网候选回退；错误信息脱敏（良好）；`useLocalOcr:true/fallbackToRemoteOcr:false` 强制；默认型号名 `gemini-3.5-flash`（Q9）；400ms 自动保存防抖。
- `localOcrTranslate.ts`：本地 OCR + 翻译编排；对冲请求错峰 700ms、预热缓存 60s、LAN 优先。
- `ocrTranslationRequest.ts`：受保护 token/术语/路径/标志检测、未译拉丁文重试、质量评估；`mergeRetryTranslations` O(n²)（B8）。
- `translationMemory.ts`：**P1 每命中即整盘读写 localStorage**；STORAGE_KEY `ysn.translationMemory.v1`，上限 1000 条，TTL 30 天。
- `detectionCandidates.ts`：原生候选 + 视觉边界检测启发式（P8）。
- `screenshotImage.ts` / `annotationGeometry.ts`：物理选区换算与标注几何，代码干净、边界 clamp 完整，无问题。
- `recordingWindows.ts`：录制控制窗口创建；大量 `[window-trace]` 日志（P5）；localStorage+事件+600ms 兜底传参（Q8）。
