1. Real‑time selection rectangle doesn’t display during drag
Symptoms: When pressing Alt + A to start a screenshot and dragging with the mouse, the selection rectangle is not visible until the mouse button is released. The selection box only updates after releasing the button, rather than following the cursor in real time.
Cause: The current implementation sets the selection rectangle only on the mouseup event. The mousemove handler does not update the rectangle state, so the overlay is only drawn once the user stops dragging.
Fix (File: useScreenshotInteraction.ts or similar hook)
Add state to track the start and current drag coordinates, and update the rectangle on each mousemove while dragging. A minimal fix looks like this:
// imports omitted …

export function useScreenshotInteraction() {
  // track dragging state
  const draggingRef = useRef(false);
  const startPointRef = useRef<{ x: number; y: number } | null>(null);
  const currentPointRef = useRef<{ x: number; y: number } | null>(null);

  // rectangle state passed to the overlay component
  const [rect, setRect] = useState<{ x: number; y: number; width: number; height: number } | null>(null);

  // start dragging on mousedown
  const handleMouseDown = (e: MouseEvent) => {
    draggingRef.current = true;
    startPointRef.current = { x: e.clientX, y: e.clientY };
    currentPointRef.current = { x: e.clientX, y: e.clientY };
    setRect(null);
  };

  // update rectangle on mousemove while dragging
  const handleMouseMove = (e: MouseEvent) => {
    if (!draggingRef.current || !startPointRef.current) return;
    currentPointRef.current = { x: e.clientX, y: e.clientY };
    const start = startPointRef.current;
    const curr = currentPointRef.current;
    setRect({
      x: Math.min(start.x, curr.x),
      y: Math.min(start.y, curr.y),
      width: Math.abs(curr.x - start.x),
      height: Math.abs(curr.y - start.y),
    });
  };

  // finish dragging on mouseup
  const handleMouseUp = () => {
    draggingRef.current = false;
    // rectangle has already been updated via mousemove; finalize capture here…
  };

  useEffect(() => {
    // attach event listeners once when the hook is used
    document.addEventListener('mousedown', handleMouseDown);
    document.addEventListener('mousemove', handleMouseMove);
    document.addEventListener('mouseup', handleMouseUp);
    return () => {
      document.removeEventListener('mousedown', handleMouseDown);
      document.removeEventListener('mousemove', handleMouseMove);
      document.removeEventListener('mouseup', handleMouseUp);
    };
  }, []);

  return { rect };
}
The UI component that renders the selection box should read the rect state and draw the overlay accordingly (e.g. via an absolutely positioned <div> or canvas). By updating rect during mousemove, the selection rectangle now follows the cursor smoothly.
If the repository uses requestAnimationFrame to throttle updates (e.g. in renderScreenshotCanvas.ts), ensure the drawing function references the live rect values on each frame.
2. Residual window remains after closing a screen recording
Symptoms: After starting a region recording via the screenshot toolbar (Alt + A → click Record (Region)) and then clicking the close button or pressing Esc, a small, blank window appears at the top left of the screen. This ghost window only disappears after quitting the app or starting another capture, which is unintuitive.
Cause: The code that creates a temporary overlay or off‑screen window for recording does not properly destroy that window when the recording is stopped. The object reference persists, so Tauri/Electron keeps it alive.
Fix (File: useScreenshotRecording.ts or similar)
Ensure the overlay window is closed and destroyed in all exit paths. Here’s a typical fix:
// assume recordingWindowRef holds the window created via Tauri/Electron

function stopRecording() {
  if (recorder) {
    recorder.stop();
    recorder = null;
  }
  // hide and destroy the overlay window if it exists
  const win = recordingWindowRef.current;
  if (win) {
    try {
      // hide immediately so the UI doesn’t linger
      win.hide();
      // give the framework a tick to process the hide before destroying
      setTimeout(() => {
        win.close();
        recordingWindowRef.current = null;
      }, 0);
    } catch (err) {
      console.error('failed to close recording window', err);
    }
  }
  // reset any state used by the screenshot toolbar …
}

// handle cancel via ESC key or toolbar button
function handleCancelRecording() {
  stopRecording();
  // also clear any drawing state and remove event listeners here
}
Add calls to stopRecording() in the ESC key handler, the toolbar’s Stop button handler, and any other code path that exits a recording session. Destroying the overlay ensures there is no persistent window when the user returns to normal mode.
