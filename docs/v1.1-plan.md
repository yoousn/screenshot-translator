# v1.1 计划：专业截图与录屏增强版

v1.1 的目标是在保持 v1.0 稳定截图翻译体验的基础上，优先补齐录屏能力，同时推进工具快捷键、OCR 可替换运行包管理和滚动截图。

## 优先级结论

1. **录屏优先**：这是 v1.1 最重要功能，先做 基础版，再逐步补齐系统声音/麦克风、区域录制、悬浮控制条。
2. **工具快捷键数字化**：低风险、高收益，已经先行落地。
3. **OCR 可替换运行包管理**：不只是压缩体积，而是把 OCR 引擎做成可替换包，后续可切换 PaddleOCR-json、RapidOCR / ONNX 或其他模型。
4. **滚动截图**：作为 beta 功能推进，先保证浏览器/普通滚动窗口可用。

## 参考 Snow Shot 后的结构结论

参考 `Snow Shot_0.7.8-beta_windows-x64-portable` 后，建议 v1.1 继续往“便携主程序 + 配置 + 插件运行包”方向演进，而不是把所有能力都塞进主 exe。

Snow Shot 的便携目录特点：

- 根目录保留主程序，另外有 `configs/`、`stores/`、`plugins/`、`pluginsDownloads/`。
- `plugins/版本号/ffmpeg/ffmpeg.exe` 独立放置，便于替换和更新。
- `plugins/版本号/rapid_ocr/*.onnx` 独立放置，OCR 模型和主程序解耦。
- `configs/functionVideoRecord.json` 单独保存录屏配置，例如帧率、编码器、最大分辨率、麦克风设备。
- `configs/systemScrollScreenshot_*.json` 单独保存滚动截图算法参数，方便后期调优。

对本项目的落地建议：

- v1.1 先保持当前用户喜欢的 `tauri-client.exe + ocr/ + ffmpeg/` 简洁便携结构。
- 后续可升级为 `plugins/<runtime-version>/ocr/`、`plugins/<runtime-version>/ffmpeg/`，同时兼容当前软件同级目录。
- 录屏配置独立保存，避免每次打开软件重新选择 FPS、分辨率、音频设备和输出目录。
- OCR / ffmpeg 下载包保留在 `pluginsDownloads/` 或 AppData 缓存，安装成功后可以清理，减少用户目录残留。
- 滚动截图参数以后也应独立成配置对象，避免写死在组件里。

## 1. 工具快捷键数字化

当前已开始落地：

| 快捷键 | 工具 |
| --- | --- |
| `1` | 方框 |
| `2` | 圆形 |
| `3` | 箭头 |
| `4` | 画笔 |
| `5` / `T` | 文字 |
| `6` | 马赛克 |

设计原则：

- 数字键只在已经框选截图区域、且不处于文字输入编辑状态时生效。
- 不拦截 `Ctrl` / `Alt` / `Shift` 组合键，避免影响复制、保存、翻译、系统快捷键。
- 工具栏 tooltip 显示快捷键，减少记忆成本。

## 2. OCR 可替换运行包管理

### 目标

把 OCR 从“固定 PaddleOCR-json 路径”升级为“可替换 OCR 运行包”。默认优先查找软件同级 `ocr\PaddleOCR-json.exe`，便携版保持 `tauri-client.exe + ocr/` 即可运行。每个 OCR 包都有自己的目录、清单、启动命令、输出适配器和模型信息。

推荐目录：

```text
ocr-runtimes/
├── paddleocr-json/
│   ├── ocr-runtime.json
│   ├── PaddleOCR-json.exe
│   ├── *.dll
│   └── models/
├── rapidocr-onnx/
│   ├── ocr-runtime.json
│   ├── rapidocr.exe 或 node/python/onnx runtime wrapper
│   └── models/
└── custom/
    └── ocr-runtime.json
```

`ocr-runtime.json` 示例：

```json
{
  "id": "paddleocr-json",
  "name": "PaddleOCR-json",
  "engine": "paddleocr-json",
  "version": "1.4.1",
  "entry": "PaddleOCR-json.exe",
  "protocol": "paddleocr-json-stdin",
  "languages": ["zh", "en", "ja", "ko"],
  "models": ["ch_PP-OCRv3_det", "ch_PP-OCRv3_rec"],
  "outputAdapter": "paddleocr-json"
}
```


`cli-json-file` 运行包示例，适合 RapidOCR / ONNX 包装器：

```json
{
  "id": "rapidocr-onnx",
  "name": "RapidOCR ONNX",
  "engine": "rapidocr",
  "version": "custom",
  "entry": "rapidocr-cli.exe",
  "protocol": "cli-json-file",
  "args": ["--image", "{image}", "--json"],
  "outputAdapter": "generic-json"
}
```

`cli-json-file` 要求运行包入口接收图片路径，并向 stdout 输出 JSON。解析器支持数组、`data`、`result`、`blocks`、`results`；文本块字段支持 `text/txt/content`、`score/confidence/conf`、`box/box_coords/points`。
### OCR 体积优化是什么

当前便携版 OCR 目录约 296 MB，主要体积来自 PaddleOCR 运行库和多语言模型：

- `mklml.dll`、`opencv_world4100.dll`、`paddle_inference.dll` 是主要大文件。
- 多语言模型包含中文、英文、日文、韩文、俄文、繁中等。

体积优化不等于一定换引擎，分三层：

1. **包管理优化**：把 OCR 包独立管理，可删除、替换、下载、切换。
2. **模型精简**：只发布常用语言模型，比如中文+英文。
3. **引擎替换**：评估 RapidOCR / ONNX，减少依赖体积。

### 对翻译速度有影响吗

- **精简模型**：基本不影响单次翻译速度；主要减少下载、解压、发布包体积。
- **可替换运行包管理**：本身不影响速度，只改变选择和加载 OCR 引擎的方式。
- **换 RapidOCR / ONNX**：可能影响 OCR 阶段速度，可能更快也可能更慢，必须实测准确率和耗时。
- **整体翻译速度**：通常由 OCR 时间 + 网络翻译时间组成。如果网络翻译接口慢，OCR 变快只会改善一部分总耗时。

### v1.1 建议落地范围

- 增加 OCR 运行包清单格式。
- 设置页显示当前 OCR 包、体积、入口文件、支持语言。
- 支持选择一个 OCR 包目录并保存为当前运行包。
- 保持 PaddleOCR-json 作为默认适配器。
- RapidOCR / ONNX 可通过 `cli-json-file` 包装器接入；真正模型包可后续独立发布。

## 3. 录屏：v1.1 最重要功能

### 基础能力

- 帧率：`30 FPS`、`60 FPS`。
- 分辨率：`480P`、`720P`、`1080P`、`原画`。
- 默认：`1080P / 30 FPS`。
- 范围：全屏录制、后续支持选区录制。
- 输出：`mp4`。
- 控制：开始、停止、录制计时、输出路径提示。

### 音频能力

录屏需要支持：

- **系统声音**：优先通过 Windows 可用音频采集设备，例如 WASAPI 默认输出 / Stereo Mix / 虚拟音频设备，具体取决于 ffmpeg 支持和系统环境。
- **麦克风**：通过 ffmpeg 音频设备列表选择麦克风。
- **混音策略**：第一版可以先提供系统声音、麦克风、静音三种模式；后续再做系统声音+麦克风混音。

### 实现路线

默认路径：录屏组件优先查找软件同级 `ffmpeg\ffmpeg.exe`，也支持在录屏页面手动选择 `ffmpeg.exe`。`resources/ffmpeg/ffmpeg.exe` 用于安装包内置。

优先接入 `ffmpeg`：

```text
录屏页面/控制入口
  -> Tauri 命令 start_recording
  -> ffmpeg 子进程
  -> stop_recording 发送停止信号/终止进程
  -> 输出 mp4 文件
```

建议先做 基础版：

1. 检测 ffmpeg 是否存在。
2. 列出可用音频设备。
3. 全屏录制，支持静音/麦克风/系统声音/系统声音+麦克风。
4. 支持 30/60 FPS 和 480P/720P/1080P/原画。
5. 提供官方 GitHub 下载/更新入口。

### 风险

- Windows 系统声音采集依赖设备和 ffmpeg 能力，不同电脑可能不同；只有检测到 WASAPI / Stereo Mix / 虚拟声卡等回环设备时才启用系统声音选项。
- 60 FPS + 原画会明显增加 CPU/GPU/磁盘压力。
- `ffmpeg.exe` 会增加发布包体积，通常 80-100 MB。

## 4. 滚动截图

基础流程：

1. 选择滚动截图区域。
2. 点击滚动截图按钮或快捷键启动。
3. 程序自动滚动目标窗口并采集多帧。
4. 根据重叠区域拼接成长图。
5. 支持手动停止、复制、保存。

难点：

- 不同软件滚动行为差异大。
- 高 DPI、多显示器、HDR、吸顶栏和动画会影响拼接。
- 第一版应作为 beta 功能，优先保证浏览器和普通滚动窗口可用。

## 当前实现说明

- 已增加录屏页和模型/视频配置页的 ffmpeg 检测、选择、官方 GitHub 下载/更新。
- 已实现 `cli-json-file` 通用 OCR 协议，可接 RapidOCR / ONNX 等自定义包装器：入口程序接收图片路径并输出 JSON 文本块。
- 已增加截图工具栏 `SCR` 滚动截图：隐藏截图层后采集 5 帧、自动滚动、按固定重叠比例拼接并复制到剪贴板。
- 滚动截图第一版偏 beta，后续需要继续做相似度重叠匹配、自动停止和可配置帧数/滚动距离。
- 已增加系统设置 → 截图识别 → 按钮间隔，用于调节截图工具栏按钮拥挤问题。

3. 录屏音频增强：系统声音、麦克风、系统声音+麦克风混音。
4. OCR 可替换运行包管理：PaddleOCR-json 清单化，并支持 `cli-json-file` 接入 RapidOCR / ONNX 包装器。
5. 滚动截图 beta。
