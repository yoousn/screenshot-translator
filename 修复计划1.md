# YSN 截图翻译 — 全面排查与修复计划

> 本文档基于 2026-05-29 对当前仓库 **真实代码** 的逐行审查整理（不依赖旧版 `修复计划0.md` 的"已修复"结论，而是重新验证现状）。
> 审查范围：`tauri-client/src/`（React 交互层）、`tauri-client/src-tauri/`（Rust 原生层与 ACL 权限）、`server/`（FastAPI / OCR / 翻译 / 缓存 / 安全）、构建与依赖配置。
> 每条问题包含：**优先级 / 问题现象 / 复现与定位 / 可能原因 / 修复建议**。

---

## 一、总体结论

整体架构（Tauri 轻客户端 + N100 FastAPI 后端 + 本地/云端双 OCR）清晰，旧版 `BUG.md` 中列出的大部分问题（端口、token 随机化、SSRF 校验、缓存命名空间、快捷键热更新、`/api/ocr` 加锁、目标语言透传等）在当前代码中确实已落地。

但本轮复审发现 **若干仍然存在、且会直接导致功能失效或在打包环境下回归的问题**，集中在两块：

1. **「钉图 (Pin)」整条链路当前必然失败** —— Tauri v2 的 ACL 权限未授权"动态创建 webview 窗口"，且能力（capability）作用域不覆盖 `pin_*` 窗口。这是 README 宣传的招牌功能，目前点击即报错。
2. **打包后（非 dev 模式）前端 `fetch` 直连后端大概率被 CORS 拦截** —— 收紧 CORS 白名单时遗漏了 Windows WebView2 的真实来源 `http://tauri.localhost`，会导致健康检查、"识字"、设置页保存/测试在正式版里失败（dev 模式因为来源是 `localhost:1420` 在白名单内，所以不易被发现）。

### 最高优先级（阻断级）问题清单

| 序号 | 问题 | 影响 |
| --- | --- | --- |
| **P0-1** | 钉图功能 ACL 权限缺失 + capability 不覆盖 `pin_*` 窗口 | 钉图点击即失败 |
| **P0-2** | CORS 白名单缺 `http://tauri.localhost`，打包后前端 fetch 被拦截 | 正式版健康检查/识字/设置保存失败 |
| **P1-1** | 整图级文本缓存键不含 `target_lang`，切目标语言命中旧译文 | 多语言翻译结果错误 |
| **P1-2** | 钉图窗口在多显示器 / 高 DPI 下位置与尺寸错位 | 贴图飞到错误位置 |
| **P1-3** | 钉图大图经 `localStorage` 传递，可能超配额导致失败/空白 | 大选区钉图失败 |

## 二、按模块拆分的修复清单

### 模块 A：钉图 (Pin) —— 当前完全不可用

  // capabilities/default.json
  "windows": ["main", "screenshot", "pin_*"],
  "permissions": [
    // ...existing...
    "core:webview:allow-create-webview-window",
    "core:window:allow-start-dragging",
    "core:window:allow-close",
    "core:window:allow-set-always-on-top"
  ]
  ```
  - 验证：框选 → 钉图 → 桌面出现置顶贴图，可拖动、滚轮缩放（若实现）、Esc/双击关闭。

#### A-2 【P1 / 中】钉图窗口位置与尺寸在多显示器 / 高 DPI 下错位
- **问题现象**：在副屏或缩放比非 100% 的显示器上钉图，贴图窗口出现在错误坐标、尺寸偏大或偏小。
- **复现 / 定位**：`handlePin()` 直接用 `rect.x/rect.y/rect.w/rect.h`（截图遮罩窗口内的 **CSS 逻辑坐标**）作为 `WebviewWindow` 的 `x/y/width/height`（相对 **整个虚拟桌面、物理像素**）。
- **可能原因**：截图遮罩窗口被定位在 `screen.x/screen.y`（物理坐标，见 `lib.rs` `start_screenshot_impl`），而选区 rect 是相对该窗口的逻辑像素，钉图时既没有叠加显示器原点偏移，也没有乘以设备像素比 `devicePixelRatio`。
- **修复建议**：钉图位置应为 `屏幕原点(物理) + rect × dpr`。建议新增一个 Rust 命令返回当前截图所用屏幕的 `(x, y, scale)`，或在前端用 `getPhysicalSelection()` 已有的物理坐标 + 屏幕原点计算，再传给窗口构造器。

#### A-3 【P1 / 中】钉图通过 `localStorage` 传图，大图可能超配额 / 残留泄漏
- **问题现象**：大选区钉图时窗口空白或报错；多次钉图后 `localStorage` 不断增大不清理。
- **复现 / 定位**：`handlePin()` 用 `localStorage.setItem(label, base64)` 传递图片，`PinPage.tsx` 用 `localStorage.getItem(label)` 读取；窗口关闭后从不 `removeItem`。
- **可能原因**：`localStorage` 单域约 5–10MB，一张高分屏 PNG 的 base64 极易超限抛 `QuotaExceededError`；且 key 持续累积。
- **修复建议**：改用 Tauri 事件 / 命令（如创建窗口后 `emit` 图片数据，或 Rust 端临时文件 + `convertFileSrc`）传图；或在 `PinPage` 读取后立即 `localStorage.removeItem(label)`，并对 `setItem` 包 try/catch 给出明确失败提示。

#### A-4 【中】「钉图 (P)」快捷键文案误导：P 键未绑定
- **问题现象**：悬浮条按钮写「钉图 (P)」，但按键盘 `P` 无任何反应。
- **复现 / 定位**：`ScreenshotPage.tsx` 的 `handleKeyDown` 只处理 `Escape / Enter / Ctrl+C / Ctrl+S / Ctrl+Q`，没有监听 `p/P`。
- **修复建议**：在 `handleKeyDown` 增加 `if (!e.ctrlKey && !e.metaKey && (e.key === "p" || e.key === "P")) { handlePin(); }`（注意避免与输入框冲突）；或直接去掉按钮文案中的 "(P)" 以免误导。

### 模块 B：客户端 ↔ 后端通信（CORS / 网络）

#### B-1 【P0 / 高】CORS 白名单缺 `http://tauri.localhost`，打包版前端 fetch 被拦截
- **问题现象**：开发模式（`npm run tauri dev`）一切正常；但 **打包成 exe 后**，控制面板长期显示「服务离线」、悬浮条「识字」失败、设置页「保存设置 / 测试连接 / 拉取模型」报网络/CORS 错误。
- **复现 / 定位**：
  - `server/app.py` 的 `CORSMiddleware` `allow_origins` 仅含 `http://localhost:1420`、`http://127.0.0.1:1420`、`tauri://localhost`、`https://tauri.localhost`。
  - 但 **Windows WebView2 下 Tauri v2 应用的真实来源是 `http://tauri.localhost`（http，非 https）**，不在白名单内。
  - 受影响的"前端直连 fetch"路径：`App.tsx` `/api/health`、`ScreenshotPage.tsx` `handleOCR` `/api/ocr`、`Settings.tsx` `/api/config/save|test|fetch_models`。（注意：`api_translate` / `api_ocr` 走 Rust reqwest，不受 CORS 影响，所以"翻译"看似正常，掩盖了问题。）
- **可能原因**：旧版 3.3「收紧 CORS」时按 macOS/Linux 的 `tauri://localhost` 与 https 形态配置，漏掉 Windows 的 http 形态。
- **修复建议**：在 `allow_origins` 增加 `http://tauri.localhost`（并保留现有项以兼容多平台与 dev）。建议同时验证打包版浏览器控制台是否还有 CORS 报错。
  - 验证方式：打包后启动，打开开发者工具看 `/api/health` 是否 200 且无 CORS 错误；或临时把 `allow_origins=["*"]` 对比现象确认根因。

### 模块 C：翻译缓存正确性

#### C-1 【P1 / 高】整图级文本缓存键不含 `target_lang`，切换目标语言命中旧译文
- **问题现象**：先把某选区翻成「中文」（结果进缓存），再到设置把目标语言改为「英语」并翻译同一画面 —— 返回的仍是中文译图。
- **复现 / 定位**：`server/image_processor.py` → `_make_text_cache_key()` 只用 `active_channel`（+ LLM 的 base_url/model）拼 namespace，**没有把 `target_lang` 纳入 key**。而 `process_and_draw` 命中 `TextTranslationCache` 后直接复用旧译文，不再调用翻译器。
- **可能原因**：旧版 1.4 修缓存时补了 model 维度，但 1.7「多目标语言」上线后没有同步把 `target_lang` 加入整图级缓存键。底层 `GLOBAL_TRANSLATE_CACHE`（单行级）已含 src/dst lang 是对的，问题只在 `image_processor` 这层整图缓存。
- **修复建议**：把目标语言透传进 `process_and_draw` 并拼入 key，例如：
  ```python
  def _make_text_cache_key(self, texts, config, target_lang="zh"):
      namespace = f"{channel}:{target_lang}"  # LLM 再附加 base_url:model
      ...
  ```
  并在 `app.py` 的 `translator_batch_fn` / `process_and_draw` 调用链把 `target_lang` 传下去。

#### C-2 【低】OCR 结果缓存键写死 `ppocr_v4`，与实际识别参数耦合不足
- **问题现象**：调整 `det_db_*` 等阈值后，旧 OCR 缓存仍可能命中（影响小）。
- **定位**：`image_processor.py` `ocr_cache_key = f"{img_md5}_ppocr_v4_{ocr_max_side}_ch_0.3_0.2_1.6"` 阈值是硬编码字符串，与实际 `_ensure_ocr` 的参数没有联动。
- **修复建议**：把真实使用的检测参数变量拼进 key（保证参数变更即缓存失效），优先级低。

### 模块 D：历史记录与状态展示

#### D-1 【中】翻译历史的 `channel` / `blocks` 字段记录不准确
- **问题现象**：历史记录页"通道"显示为目标语言或空，"识别块"恒为 1。
- **复现 / 定位**：`ScreenshotPage.tsx` `handleTranslate` 写历史：
  ```ts
  blocks: 1, // Optional: ...
  channel: configRef.current.channel || configRef.current.targetLang || "auto",
  ```
  `channel` 混用了 `targetLang`，且 `config.channel` 是客户端展示值（常为 undefined），真实通道在服务器；`blocks` 写死 1。
- **修复建议**：`/api/translate` 响应头已返回 `X-Trace-Translate-Units`（块数）与 `X-Translate-Texts`；可在 Rust `api_translate` 一并回传块数与真实通道（或读 `/api/config/current` 的 `active_channel`），前端据此写入历史。至少先去掉误导：`blocks` 用 `translatePairs.length`，`channel` 用真实通道字段。

#### D-2 【中】全局快捷键失败提示文案硬编码「Alt + A」，与可配置快捷键不符
- **问题现象**：用户把截图热键改成别的组合并保存，若该热键被占用注册失败，弹出的通知仍写「Alt + A 注册失败」。
- **复现 / 定位**：`App.tsx` `checkShortcutStatus` 的 `notification.error` description 写死 `Alt+A`；而 `lib.rs` 已支持读取 `config.hotkey` 注册，`get_shortcut_status` 返回的 `e`（`shortcutError`）其实已含真实键值与原因。
- **修复建议**：通知文案改用 `shortcutError`（或读取当前 `config.hotkey`）动态展示，避免误导。

#### D-3 【低】设置页未同步服务器真实激活通道
- **问题现象**：首次安装 / 本地 `config.json` 为空时，设置页「活动翻译信道」显示 google，但服务器 `active_channel` 可能是别的（旧版 1.1 遗留的展示层）。
- **定位**：`Settings.tsx` `loadSettings` 只回填本地 `config.json`，未调用已存在的 `GET /api/config/current`。
- **修复建议**：`loadSettings` 额外拉 `/api/config/current`，用其 `active_channel` 校正 `channel` 字段显示。

### 模块 E：后端健壮性与工程化（低优先，非阻断）

#### E-1 【低】启动时 token 用 `print` 而非 `logger`，与"全面 logging 化"目标不一致
- **定位**：`app.py` 顶部 `print(f"[Security] 当前 client_token: ...")`。属于有意保留（确保运维一定看到），可接受；若要统一可改 `logger.warning`。

#### E-2 【低】`/api/translate` 为同步 `def`，大图在高并发下占用线程池
- **定位**：`app.py` `translate_image` 是同步函数，FastAPI 会丢线程池执行；OCR 本身有 `_ocr_infer_lock` 串行化。当前单机自用无碍，多客户端高并发时吞吐受限。可作为后续异步化/队列化的观察点。

#### E-3 【低】`quick_fullscreen_capture` 固定取 `screens[0]`
- **定位**：`lib.rs` `quick_fullscreen_capture` 总是截主屏，与 `start_screenshot` 的"光标所在屏"策略不一致。多屏用户用"截全屏"可能截错屏。

## 四、需要你确认 / 补充的信息

1. **打包目标平台**：确认正式版主要跑 Windows（决定 CORS 是否必须加 `http://tauri.localhost`，以及钉图坐标的 DPI 处理方式）。
2. **钉图传图方式偏好**：A-3 是否接受改用 Tauri 事件/临时文件替代 `localStorage`（推荐），还是先用 try/catch + 清理做最小修复。
3. **多目标语言是否已实际启用**：C-1 修复需把 `target_lang` 透传进缓存键，请确认目标语言切换是当前真实使用场景（影响优先级判定）。
4. **CORS 根因复现**：能否提供一次"打包版"运行时浏览器控制台/服务端日志，以最终确认 B-1（dev 模式无法复现该问题）。
